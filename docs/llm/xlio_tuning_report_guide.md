# XLIO Tuning Report Guide

> This document is designed for LLM chatbot consumption. It teaches you how to
> analyze XLIO tuning reports and recommend configuration changes.
> This document is self-contained ‚Äî every config knob referenced includes its
> full path, default value, and what it controls.

## Purpose & Usage

This document helps you analyze post-run diagnostic reports generated by XLIO
(a network acceleration library). Customers will paste a tuning report and ask
you to suggest configuration improvements.

**Your analysis workflow:**
1. Check report completeness ‚Äî is it full detail or fallback detail? (see next section)
2. Check the `## Effective Config` section ‚Äî note what's already non-default and
   check the `reason:` field for each entry (`User-configured`, `Profile`, or
   `Auto-corrected`) to understand why it was changed
3. If you previously generated a config for this customer, compare it against the
   Effective Config section. Use the `reason:` field to identify discrepancies ‚Äî
   if a value you recommended shows `reason: Profile` instead of
   `reason: User-configured`, the profile overrode it. If it shows
   `reason: Auto-corrected`, a system constraint forced a different value.
   Investigate discrepancies before deep-diving into WARNINGs.
4. Scan for `# WARNING` annotations ‚Äî these are the anomalies to address
5. If WARNINGs present: look up each one in the Troubleshooting Rules below
6. If NO WARNINGs: use the "No WARNINGs ‚Äî Throughput Below Expectations" checklist
7. Cross-reference config knobs with the Config Knob Quick Reference in this document
8. Ask the customer clarifying questions before recommending changes

**Important context you don't have from the report alone:**
- What throughput/latency the customer expected vs achieved
- Whether the workload is latency-sensitive or throughput-sensitive
- Network topology (switch hops, MTU mismatches between endpoints)
- Peer configuration (is the other end also running XLIO?)
- Whether the application is CPU-bound or network-bound

**Always ask the customer:**
- "What throughput/latency did you expect, and what did you measure?"
- "Is this a latency-sensitive or throughput-sensitive workload?"
- "How many connections, and are they short-lived or long-lived?"

**Report generation:** The report is generated at process exit. Enable it with:
- `monitor.report.mode: enable` ‚Äî always generate
- `monitor.report.mode: auto` ‚Äî generate only when anomalies are detected (default)
- `monitor.report.mode: disable` ‚Äî never generate

**Important `auto` mode behavior:** In `auto` mode, the report is only generated
when specific anomalies are detected:
- Buffer pool allocation failures for `rx_rwqe`, `rx_stride`, and `tx` pools
- HW RX packet drops
- TX WQE exhaustion (`ring_tx_dropped_wqes > 0`)

**Not checked by `auto` mode:** `rx_ptr` and `zc` buffer pool failures,
`tcp_seg_pool_alloc_failures`, software RX drops, retransmits, and offload
ratio issues do NOT trigger auto-generation. A customer with only these
anomalies will get no report in `auto` mode.

A customer who explicitly sets `mode: enable` may get a clean report with no
WARNINGs ‚Äî they are proactively tuning. Use the "No WARNINGs" checklist for
this case.

**How to present recommendations:**
When recommending config changes, always present them as a before/after diff
showing the config knob, its current value (from Effective Config or default),
and the recommended value. Use JSON config format so the customer can directly
apply it. Example:

```
Recommended changes (add to your XLIO JSON config file):
{
  "core": {
    "resources": {
      "memory_limit": "8 GB"       // was: 2 GB (default)
    }
  },
  "performance": {
    "rings": {
      "rx": {
        "ring_elements_count": 256  // was: 128 (striding RQ default)
      }
    }
  }
}
```

If the customer is using `XLIO_INLINE_CONFIG`, show the equivalent inline format.
Always explain *why* each change is recommended (link back to the WARNING or
checklist item that triggered it).

**Privacy note:** Reports contain hostnames, NIC names, command lines, and
config values. If the customer is sharing reports through a cloud-hosted
chatbot, advise them to redact hostnames and internal IP addresses if privacy
is a concern.

## Report Format & Completeness

### Report Sections (in order)
The report always contains these sections in this order:
1. **Preamble** ‚Äî format version, timestamp, PID, duration
2. **System Context** ‚Äî XLIO version, kernel, NIC devices, hugepages
3. **Active Profile** ‚Äî which built-in profile is active
4. **Effective Config** ‚Äî non-default config values only
5. **Runtime Stats** ‚Äî traffic counters, errors, ring diagnostics, buffer pools
6. **Socket Summary** ‚Äî socket counts, offload status, listen stats
7. **Performance Indicators** ‚Äî derived metrics with thresholds

### Effective Config Format
Each non-default parameter is printed as:
```
key: value
  # default: default_value | reason: <reason> | Human-Readable Title
```
The `reason` field tells you **why** the value differs from the default:
- `User-configured` ‚Äî the customer explicitly set this in their JSON config or `XLIO_INLINE_CONFIG`
- `Profile` ‚Äî set by the active built-in profile (e.g., `latency`, `nginx`)
- `Auto-corrected` ‚Äî the system adjusted the value due to a constraint (e.g., "Capped by CQ size", "Forced when TSO enabled"). The parenthetical description explains the specific constraint.

The `reason` field is omitted when the parameter has no recorded change reason
(e.g., nested config objects). Parameters at their default values are not shown.

Example ‚Äî user-configured value:
```
core.resources.memory_limit: 8 GB
  # default: 2 GB | reason: User-configured | Memory limit for XLIO resources
```
Example ‚Äî profile-set value:
```
performance.polling.iomux.poll_usec: -1
  # default: 100000 | reason: Profile | Polling RX duration (¬µsec)
```
Example ‚Äî auto-corrected value:
```
performance.rings.rx.qp_compensation_level: 128
  # default: 8192 | reason: Auto-corrected (Capped by CQ size) | QP compensation level
```
If you see `# All parameters at default values`, no config has been customized.
If you see `# Config registry not available`, the config registry was not
initialized ‚Äî this happens when using legacy environment variables (e.g.,
`XLIO_MEMORY_LIMIT`) without the new JSON config system. The Effective Config
section will be empty. Recommend the customer switch to JSON config
(`XLIO_CONFIG_FILE` or `XLIO_INLINE_CONFIG`) for full diagnostic support.

### Two Report Detail Levels

**CRITICAL:** The report has two fundamentally different detail levels. You MUST
detect which detail level the customer's report is in before applying troubleshooting rules.

**Full detail** (per-socket stats available):
- Shows: `total_rx_packets`, `total_tx_packets`, `total_rx_bytes`, `total_tx_bytes`,
  `rx_throughput`, `tx_throughput`, `sw_rx_packets_dropped`, `tx_errors`, `rx_errors`,
  `tx_retransmits`, `poll_hit_rate`, `sw_rx_drop_rate`, `tx_retransmit_rate`,
  offload traffic split, listen connection stats.
- All 21 WARNINGs can fire.
- **How to detect:** The annotation
  `# Per-socket traffic stats require monitor.stats.fd_num > 0` is **absent** from
  Runtime Stats, and per-socket fields (`total_rx_packets`, `tx_errors`, etc.) are
  present.

**Fallback detail** (no per-socket stats, ring-only):
- Shows: Only ring-level totals ‚Äî `ring_total_rx_packets`, `ring_total_tx_packets`,
  `ring_total_rx_bytes`, `ring_total_tx_bytes`, `ring_total_tx_retransmits`.
- Only 12 of 21 WARNINGs can fire (resource exhaustion, HW drops, ring diagnostics).
  Missing: `tx_errors`, `rx_errors`, `sw_rx_*_dropped`, `sw_rx_drop_rate`,
  `tx_retransmit_rate`, `poll_hit_rate`, non-offloaded traffic ratio,
  `listen_conn_dropped`.
- **How to detect:** The Runtime Stats section contains the annotation
  `# Per-socket traffic stats require monitor.stats.fd_num > 0`.

**If the customer's report is fallback detail**, your first recommendation should be:
"Re-run with `monitor.stats.fd_num` set to at least the number of concurrent
sockets (e.g., `monitor.stats.fd_num: 100`) to get full diagnostic data."

**Edge case ‚Äî no sockets created:**
If Runtime Stats or Socket Summary show `# No sockets were created`, the process
ran under XLIO but never created any TCP/UDP sockets. This typically means:
- The application exited too quickly (e.g., startup error)
- XLIO was loaded via `LD_PRELOAD` but the command doesn't use sockets (e.g., `ls`)
- The application uses non-offloadable socket types only (Unix domain sockets)
There is nothing to tune ‚Äî ask the customer to verify XLIO is loaded with the correct application.

### Partial Coverage Warning
Even in full detail, if the stats pool is smaller than the total socket count,
the report shows:
```
  # Note: per-socket traffic stats cover X/Y sockets (increase monitor.stats.fd_num for full coverage)
```
This means traffic split numbers (offloaded vs non-offloaded bytes) are
**underestimates** ‚Äî they only cover X of the Y sockets. Recommend increasing
`monitor.stats.fd_num` to at least Y.

### Report Completeness
The last lines of every report are:
```
# End of XLIO Tuning Report
# Report generated successfully
```

- **Both lines present** ‚Äî all sections generated without errors.
- **`# End of XLIO Tuning Report` present but `# Report generated successfully`
  missing** ‚Äî one or more sections threw an exception during generation. Look for
  `# ERROR:` annotations above to identify what failed.
- **Neither line present** ‚Äî the process crashed during report generation and the
  file is truncated.

**If the report is incomplete or truncated**, tell the customer: "Your report
appears incomplete ‚Äî it may have been cut short by an error or crash during
generation. Please re-run and verify the report file is complete before
analysis."

### Report Format Version
The preamble is entirely `#`-prefixed comment lines. The version appears as:
`# report_format_version: 1`
This document covers version 1. If you see a higher version number, some fields
may have been added, removed, or renamed ‚Äî proceed with caution and note the
version mismatch to the customer.

### Annotations Reference
The report uses three types of inline annotations:

**`# WARNING:` annotations** ‚Äî Anomalies. Each one has a troubleshooting rule below.

**`# ERROR:` annotations** ‚Äî Report generation failures. If you see
`# ERROR: report generation failed: <message>` (or `unknown exception`), one or
more report sections failed to generate. The report may still end with
`# End of XLIO Tuning Report` but will be missing data. Tell the customer:
"Your report contains a generation error ‚Äî some sections are missing. Please
re-run and check for crashes during XLIO shutdown."

**`# Note:` annotations** ‚Äî Contextual information, not problems:
- `# Note: lifetime average, process ran < 5min` ‚Äî Throughput values include
  startup overhead (device probing, memory registration, TCP slow-start).
  Steady-state throughput is likely higher. For benchmarks, recommend running > 5 min.
- `# Note: 0 accepted with established > 0 is expected when using event-driven API (poll groups) instead of accept()` ‚Äî The application uses XLIO Ultra API / poll groups
  which bypasses accept(). This is normal, not a problem.
- `# event-driven API (poll groups) ‚Äî poll counters not applicable` ‚Äî Shown when
  `poll_hit_rate` is `N/A`. The application uses poll groups / XLIO Ultra API which
  bypasses the poll/recv loop, so poll hit rate is not a meaningful metric. Not a problem.
- `# Note: per-socket traffic stats cover X/Y sockets` ‚Äî See Partial Coverage above.
- `# Per-socket traffic stats require monitor.stats.fd_num > 0` ‚Äî Fallback detail indicator.

## How to Analyze a Report

### Step 0: Check Report Completeness and Duration
Before analyzing, check the duration and determine the report's detail level:

**Check duration first:** Look at `Duration:` in the Preamble.
- If Duration is in **milliseconds** (e.g., `Duration: 50ms`), the process exited
  almost immediately. There is nothing meaningful to tune ‚Äî counters are near zero
  and throughput is meaningless. Ask the customer: "Your process ran for less than
  a second. Please verify the application ran successfully and re-run for at least
  5 minutes for meaningful diagnostics."
- If Duration is **< 5 minutes**, throughput values include startup overhead. The
  report will annotate this with `# Note: lifetime average, process ran < 5min`.

**Determine detail level:**
- If you see `total_rx_packets`, `tx_errors`, etc. ‚Üí **Full detail**. Proceed normally.
- If you see `ring_total_rx_packets` and `# Per-socket traffic stats require monitor.stats.fd_num > 0` ‚Üí **Fallback detail**. Recommend re-running with `monitor.stats.fd_num` set appropriately before deep analysis.
- If you see `# Note: per-socket traffic stats cover X/Y sockets` ‚Üí **Partial coverage**. Note that traffic split numbers may be underestimates.

### Step 1: Read System Context
Check `nic_device` lines for NIC speed and MTU. A 400 Gbps NIC with MTU 1500
has very different tuning needs than a 100 Gbps NIC with MTU 9000 (jumbo frames).
Check `hugepages_*_free` ‚Äî if 0, memory allocation is degraded.

**Multiple NICs:** If the report shows multiple `nic_device` lines with different
speeds or MTUs, ask the customer which NIC carries the primary data traffic.
Focus tuning recommendations on that NIC. A common pattern is one high-speed
NIC (e.g., 100/200/400 Gbps) for data and one lower-speed NIC for management
‚Äî tuning the management NIC is not useful.

### Step 2: Check the Active Profile
The `profile_spec` field shows which built-in profile is active. Profiles set
many config values at once. If `profile_spec: none`, the customer may benefit
from selecting one (`profiles.spec`, default: `none`). Available profiles:
- `latency` ‚Äî trading / real-time control
- `ultra_latency` ‚Äî lowest latency, 100% CPU polling
- `nginx` ‚Äî HTTP proxy / load balancer workloads
- `nginx_dpu` ‚Äî nginx running on NVIDIA DPU
- `nvme_bf3` ‚Äî SPDK solution over NVIDIA DPU BF3

**Profile settings and the `reason` field.** When a profile is active, many
config knobs are set to non-default values. The Effective Config section tags
each entry with its provenance:
- `reason: Profile` ‚Äî set by the active profile. Don't recommend a change the
  profile already makes. To override a profile setting, tell the customer
  explicitly (e.g., "Your nginx profile sets TSO to ON, but...").
- `reason: User-configured` ‚Äî the customer explicitly set this. Ask why before
  recommending a different value.
- `reason: Auto-corrected (...)` ‚Äî system constraint forced a change. The fix
  may be upstream (e.g., increasing CQ size to stop capping a value).

**Profile quick reference** (key distinguishing traits only):

| Profile | Focus | Key overrides |
|---|---|---|
| `ultra_latency` | Lowest latency, 100% CPU | poll=-1 (‚àû), TSO=OFF, tcp_nodelay, gro=disabled |
| `latency` | Low latency | poll=-1 (‚àû), TSO=OFF, tcp_nodelay, gro=disabled |
| `nginx` | HTTP proxy throughput | TSO=ON, rx_poll_on_tx_tcp, rx_cq_wait_ctrl, tcp_3t_rules |
| `nginx_dpu` | nginx on DPU | Same as nginx, lower memory |
| `nvme_bf3` | SPDK on BF3 | TSO=ON, LRO=ON, strides=8192, tcp_nodelay |

### Step 3: Review Effective Config
This section shows ONLY non-default values. Format is:
```
key: value
  # default: default_value | reason: <reason> | Title
```
**Use the `reason` field to guide your analysis:**
- `reason: User-configured` ‚Äî the customer chose this. Ask why before changing it.
- `reason: Profile` ‚Äî set by the active profile. Don't recommend what the profile already does.
- `reason: Auto-corrected (description)` ‚Äî system constraint forced the change. The fix may be upstream (e.g., increase CQ size to stop capping ring elements).

Look for:
- Memory limits that may be too low for the workload
- Hardware features (TSO, LRO) that could help but aren't enabled
- Polling settings that don't match the workload pattern
- Auto-corrected values whose constraints could be relaxed

If you see `# Config registry not available`, the JSON config system is not
activated ‚Äî see Report Format section above for details.

### Step 4: Scan for WARNINGs in Runtime Stats
Every `# WARNING` annotation indicates a detected anomaly. Look up each one
in the Troubleshooting Rules section below. Multiple WARNINGs often have a
common root cause ‚Äî address them together.

**When multiple WARNINGs are present:** Collect all recommendations and "ask
the user" questions from all triggered rules first, then **deduplicate** before
presenting to the customer. Common patterns:
- `buffer_pool_*_alloc_failures` + `hw_rx_packets_dropped` + `ring_tx_dropped_wqes`
  all point to `core.resources.memory_limit` ‚Äî recommend it once, not three times.
- `tx_retransmits` + `tx_retransmit_rate` + `ring_tls_*_resyncs` share a root
  cause (network loss) ‚Äî present one consolidated diagnosis.
- Multiple rules ask about CPU utilization, memory, or peer config ‚Äî ask each
  question once.

**If there are NO WARNINGs:** Skip to the "No WARNINGs ‚Äî Throughput Below
Expectations" section for a structured tuning checklist.

### Step 5: Check Socket Summary & Detect Protocol
Verify all sockets are offloaded. Non-offloaded sockets bypass XLIO entirely
and use the slow kernel path.

**Detect UDP vs TCP workloads:** Check `tcp_sockets` and `udp_sockets` counts.
- If `udp_sockets > 0` and `tcp_sockets == 0` ‚Üí **UDP-only workload**. Skip all
  TCP-specific rules (retransmits, TSO, LRO, TLS, `tcp_seg_pool`, congestion
  control). The "No WARNINGs" checklist items for TSO, LRO, and TCP CC also
  do not apply.
- If both are > 0 ‚Üí **mixed workload**. Apply TCP rules only to TCP-related
  WARNINGs.
- UDP workloads benefit most from: ring sizes, memory headroom, MTU/jumbo
  frames, CPU affinity, and polling configuration.

### Step 6: Check Performance Indicators
These are derived metrics with built-in thresholds:
- `poll_hit_rate` < 80% ‚Üí polling inefficiency
- `sw_rx_drop_rate` > 0.01% ‚Üí application not consuming fast enough
- `tx_retransmit_rate` > 0.1% ‚Üí network issues
- `hw_rx_packets_dropped` > 0 ‚Üí NIC ring overflow

If `poll_hit_rate: N/A` with a note about event-driven API, this metric is
not applicable ‚Äî the application uses poll groups / XLIO Ultra API. Not a problem.

### Step 7: Sanity-Check the Numbers
Before making recommendations, verify the data makes sense:
- **Average packet size:** `total_rx_bytes / total_rx_packets` should be between
  64 and 9000 bytes. If it's outside this range, per-socket uint32 counters may
  have overflowed before aggregation (see Limitations).
- **Throughput vs NIC speed:** If `rx_throughput` shows 50 Gbps on a 100 Gbps NIC
  but the `# Note: lifetime average, process ran < 5min` annotation is present,
  steady-state throughput is likely higher.
- **Ring vs socket traffic:** When both are present, `ring_total_tx_packets` >
  `total_tx_packets` is normal (ring includes TCP ACKs). But `ring_total_rx_bytes`
  should ‚âà `total_rx_bytes`. Large discrepancies warrant investigation.

## Troubleshooting Rules

> For each WARNING the report can produce, this section explains the root cause,
> what to cross-check, and which config knobs to adjust.
>
> **Protocol applicability:** Rules marked **(TCP only)** do not apply to UDP-only
> workloads. Check the Socket Summary for `tcp_sockets` vs `udp_sockets` counts
> before applying TCP-specific rules.

### Resource Exhaustion

#### hugepages fully consumed
**WARNING:** `hugepages_<size>kB_free: 0 # WARNING: hugepage pool fully consumed`

**What it means:** All pre-allocated hugepages are in use. XLIO falls back to
regular 4KB pages, causing TLB misses and reduced throughput.

**What to check:**
- `core.resources.memory_limit` ‚Äî if set very high, XLIO consumes more hugepages
- `core.resources.hugepages.enable` ‚Äî should be `true` (default)
- `core.resources.hugepages.size` ‚Äî check which hugepage size is configured

**Config fix:**
- Not a config fix ‚Äî this is a system-level issue. The customer needs to allocate
  more hugepages at the OS level:
  `echo <N> > /sys/kernel/mm/hugepages/hugepages-2048kB/nr_hugepages`
- If the customer cannot allocate more hugepages, reduce `core.resources.memory_limit`
  to lower XLIO's memory footprint.

**Ask the user:** "How many hugepages are allocated on your system? How many other
processes share them? Can you increase the allocation?"

---

#### Buffer pool allocation failures
**WARNING:** `buffer_pool_<type>_alloc_failures: <N> # WARNING: allocation failures`

**What it means:** XLIO ran out of pre-allocated buffers and had to drop or stall
operations. The pool type tells you which path is affected:
- `rx_rwqe` ‚Äî receive Work Queue Element buffers (standard RX path): incoming packets
  may be dropped because the NIC has no buffers to write received data into
- `rx_stride` ‚Äî striding RQ buffers (used when `hardware_features.striding_rq.enable: true`,
  which is the default): same effect as `rx_rwqe` but for the strided receive path
- `rx_ptr` ‚Äî receive pointer buffers (used for scatter-gather RX): incoming packets
  may be dropped when the pointer pool is exhausted
- `tx` ‚Äî transmit path: sends may fail (contributes to `tx_errors`)
- `zc` ‚Äî zero-copy path: zero-copy send/receive operations stalled

**What to check:**
- `core.resources.memory_limit` ‚Äî this caps total buffer allocation
- `hugepages_*_free` ‚Äî if 0, allocation failures may be due to hugepage exhaustion
- The corresponding `buffer_pool_<type>_size` ‚Äî shows how many buffers were allocated

**Config fix:**
- Increase `core.resources.memory_limit` (e.g., from 2 GB to 4 GB or 8 GB)
- If hugepages are exhausted, also increase OS hugepage allocation
- For RX pools: increasing `performance.rings.rx.ring_elements_count` or
  `performance.rings.rx.spare_buffers` can help absorb traffic bursts
- For TX pools: check if `performance.rings.tx.ring_elements_count` is too small

**Ask the user:** "What is your current `core.resources.memory_limit`? How much
total system RAM is available? Are other XLIO processes running on the same machine?"

---

#### TCP segment pool exhaustion **(TCP only)**
**WARNING:** `tcp_seg_pool_alloc_failures: <N> # WARNING: segment pool exhaustion (TX stalls)`

**What it means:** The TCP segment pool ran out of segments. TCP sends stall until
segments are freed. This directly causes TX latency spikes and throughput drops.

**What to check:**
- `tcp_seg_pool_size` ‚Äî shows the pool size (higher = more concurrent TCP segments)
- `core.resources.memory_limit` ‚Äî the pool size is bounded by available memory
- Number of connections ‚Äî many connections consume segments simultaneously

**Config fix:**
- Increase `core.resources.memory_limit` to allow a larger segment pool
- Reduce `performance.buffers.tcp_segments.socket_batch_size` if sockets are
  hoarding segments (reduces per-socket reservation, frees segments for others)

**Ask the user:** "How many concurrent TCP connections does your application maintain?
Is traffic bursty or steady-state?"

---

#### WQE exhaustion
**WARNING:** `ring_tx_dropped_wqes: <N> # WARNING: WQE exhaustion detected`

**What it means:** The NIC's send queue (Work Queue Elements) overflowed. Sends
were dropped at the hardware level. This is the most direct cause of TX throughput loss.

**What to check:**
- `tx_errors` ‚Äî should correlate with `ring_tx_dropped_wqes` (every dropped WQE = one TX error)
- `buffer_pool_tx_alloc_failures` ‚Äî if also non-zero, memory is the root cause
- `performance.rings.tx.ring_elements_count` ‚Äî controls send queue depth
- `core.resources.memory_limit` ‚Äî caps total WQE allocation

**Config fix:**
- Increase `core.resources.memory_limit` (primary fix ‚Äî allows more WQEs)
- Increase `performance.rings.tx.ring_elements_count` (directly sizes the send queue)
- Enable TSO (`hardware_features.tcp.tso.enable: enable`) ‚Äî TSO reduces the number
  of WQEs needed by coalescing multiple segments into one send operation

**Ask the user:** "What is the message send pattern? Many small sends, or fewer
large sends? Is TSO currently enabled?"

### TX Issues

#### TX errors
**WARNING:** `tx_errors: <N> # WARNING: TX errors detected`

**What it means:** Socket-level send operations failed with an error other than
EAGAIN (which is normal backpressure). These are genuine failures where data
was not transmitted.

**What to check ‚Äî use these fields to identify the root cause:**
- `ring_tx_dropped_wqes` ‚Äî if > 0 and ‚âà `tx_errors`: WQE exhaustion is the cause (see Resource Exhaustion)
- `ring_tx_dropped_wqes` = 0: errors are from the TCP stack, not the NIC
  - Check `tcp_seg_pool_alloc_failures` ‚Äî segment pool exhaustion causes ENOMEM
  - Check `buffer_pool_tx_alloc_failures` ‚Äî buffer exhaustion causes ENOMEM
- If none of the above are non-zero: errors may be connection resets (peer closed),
  or the application is sending on a closed socket

**Config fix:**
- If caused by WQE exhaustion: see `ring_tx_dropped_wqes` rule above
- If caused by segment/buffer exhaustion: increase `core.resources.memory_limit`
- If caused by connection resets: not a config issue ‚Äî application or peer behavior

**Ask the user:** "Are you seeing connection resets or timeouts at the application level?
Is the peer endpoint healthy?"

---

#### TX retransmits **(TCP only)**
**WARNING:** `tx_retransmits: <N> # WARNING: TCP retransmits (congestion or packet loss)`
**WARNING:** `tx_retransmit_rate: <pct>% # WARNING: high retransmit rate` (threshold: > 0.1%)
**WARNING:** `ring_total_tx_retransmits: <N> # WARNING: retransmits detected` (fallback detail)

**What it means:** TCP segments had to be retransmitted because the receiver didn't
acknowledge them in time. This indicates network congestion, packet loss, or
receiver-side drops.

**What to check:**
- `hw_rx_packets_dropped` on the **receiver side** ‚Äî receiver may be dropping packets
- Network path: switch congestion, cable issues, MTU mismatch
- `network.protocols.tcp.congestion_control` ‚Äî which CC algorithm is in use:
  - `lwip` (default) ‚Äî XLIO's built-in lightweight TCP CC, suitable for most workloads
  - `disable` ‚Äî no congestion control; only appropriate on lossless fabrics (e.g., RoCE
    with PFC) where the network guarantees no packet loss. Using `disable` on a lossy
    network will cause catastrophic retransmit storms.
- `network.protocols.tcp.timer_msec` ‚Äî TCP timer granularity affects retransmit detection

**Config fix:**
- Retransmits are rarely fixed by XLIO config alone ‚Äî they indicate network issues
- If the receiver is also running XLIO, check its report for `hw_rx_packets_dropped`
  and `sw_rx_packets_dropped`
- Consider `network.protocols.tcp.timestamps: enable` for more accurate RTT estimation
- If using jumbo frames, verify MTU matches end-to-end (check `nic_device` MTU in both reports)
- On lossless fabrics with PFC, `network.protocols.tcp.congestion_control: disable`
  eliminates CC overhead ‚Äî but **only** recommend this if the customer confirms a
  lossless network. On lossy networks, this makes things worse.

**Ask the user:** "Is the peer also running XLIO? Can you share its tuning report?
Are there switches between the endpoints? What is the MTU on both sides?"

### RX Issues

#### RX errors
**WARNING:** `rx_errors: <N> # WARNING: RX errors detected`

**What it means:** Receive operations failed at the socket level. Less common than
TX errors ‚Äî typically indicates protocol violations or corrupted packets.

**Config fix:** Not typically config-fixable. Investigate network health.

**Ask the user:** "Are you seeing any errors in `dmesg` or NIC firmware logs?"

---

#### Software RX drops
**WARNING:** `sw_rx_packets_dropped: <N> # WARNING: non-zero drops`
**WARNING:** `sw_rx_bytes_dropped: <N> # WARNING: non-zero drops`
**WARNING:** `sw_rx_drop_rate: <pct>% # WARNING: non-zero drop rate` (threshold: > 0.01%)

**What it means:** XLIO received packets from the NIC successfully but the application
didn't consume them fast enough. Packets were dropped in software receive buffers.
This is the #1 cause of throughput loss in receive-heavy workloads.

**What to check:**
- Is the application CPU-bound? (check if CPU is at 100%)
- `poll_hit_rate` ‚Äî if low, the application isn't polling frequently enough
- `performance.polling.rx_poll_on_tx_tcp` ‚Äî if disabled, TX-heavy periods skip RX polling
- `buffer_pool_rx_*_alloc_failures` ‚Äî if non-zero, buffers are also exhausted
- `performance.override_rcvbuf_limit` ‚Äî receive buffer limit

**Config fix:**
- If CPU-bound: not a config fix ‚Äî application needs optimization or more CPU cores
- If poll_hit_rate is low: see the poll_hit_rate rule below
- Enable `performance.polling.rx_poll_on_tx_tcp: true` if the workload is bidirectional
- Increase `performance.override_rcvbuf_limit` to allow larger receive buffers
- Enable LRO (`hardware_features.tcp.lro: enable`) to reduce per-packet CPU cost
- If using `profile_spec: none`, consider `latency` or `ultra_latency` profile for
  more aggressive polling

**Ask the user:** "Is your application CPU-bound during the test? How many CPU cores
are dedicated to XLIO traffic? Is the workload bidirectional or receive-only?"

---

#### Hardware RX drops
**WARNING:** `hw_rx_packets_dropped: <N> # WARNING: HW drops detected`

**What it means:** The NIC itself dropped packets before XLIO could process them.
The NIC's receive ring buffer overflowed. This is more severe than software drops ‚Äî
the packet never reached XLIO at all.

**What to check:**
- `performance.rings.rx.ring_elements_count` ‚Äî receive ring depth
- `performance.rings.rx.spare_buffers` ‚Äî spare buffers for burst absorption
- `buffer_pool_rx_*_alloc_failures` ‚Äî if non-zero, XLIO couldn't replenish the ring
- `core.resources.memory_limit` ‚Äî may be too low for the traffic rate

**Config fix:**
- Increase `performance.rings.rx.ring_elements_count` (e.g., double it)
- Increase `performance.rings.rx.spare_buffers`
- Increase `core.resources.memory_limit` if buffer pool failures are also present
- Enable LRO (`hardware_features.tcp.lro: enable`) ‚Äî reduces packets-per-second
  at the NIC level by coalescing multiple TCP segments
- For striding RQ workloads: increase `hardware_features.striding_rq.strides_num`

**Ask the user:** "What is the incoming packet rate (packets per second)?
What is the average packet size? Is the sender using TSO (which produces large packets)
or sending many small packets?"

### Offload Issues

#### Non-offloaded sockets
**WARNING:** `# WARNING: <N>/<N> sockets are non-offloaded`

**What it means:** Some sockets are going through the kernel network stack instead
of XLIO. These sockets get zero acceleration benefit.

**What to check:**
- `acceleration_control.default_acceleration` ‚Äî if `false`, sockets must be
  explicitly opted-in via rules
- `acceleration_control.rules` ‚Äî if rules are defined, they may exclude some sockets
- Socket type: XLIO only offloads TCP and UDP over IPv4/IPv6. Unix domain sockets,
  raw sockets, etc. are never offloaded.
- Multicast: multicast UDP may not be offloaded depending on configuration

**Config fix:**
- Set `acceleration_control.default_acceleration: true` (default) to offload all
  eligible sockets
- Review `acceleration_control.rules` ‚Äî remove rules that exclude target sockets
- If using nginx profile: ensure `applications.nginx.workers_num` is set correctly

**Ask the user:** "What types of sockets does your application create?
Are any sockets intentionally non-offloaded (e.g., management/control connections)?"

---

#### High non-offloaded traffic ratio
**WARNING:** `# WARNING: <pct>% of RX bytes went through non-offloaded path`

**What it means:** More than 50% of received bytes went through the slow kernel path.
The application is running but XLIO is not accelerating the primary traffic.

**What to check:**
Same as "Non-offloaded sockets" above, plus:
- `total_rx_os_packets` / `total_tx_os_packets` ‚Äî these count packets that went
  through the OS kernel path. High OS packet counts relative to total packets
  confirm the offload issue, even if the non-offloaded traffic ratio is just
  below the 50% WARNING threshold.
- The application may be creating sockets before XLIO is loaded (LD_PRELOAD timing)
- The traffic may be on ports/addresses not matched by acceleration rules

**Config fix:**
Same as "Non-offloaded sockets" above.

**Ask the user:** "How is XLIO loaded ‚Äî via LD_PRELOAD or linked directly?
Which ports and addresses carry your primary traffic?"

### Hardware Offload Validation

**Build note:** TLS-related fields (`ring_tls_rx_resyncs`, `ring_tls_tx_resyncs`,
`ring_tls_rx_auth_fail`, `tls_tx_bytes`, `tls_rx_bytes`) only appear in XLIO builds
with uTLS support (`DEFINED_UTLS`). If you don't see any TLS fields in a report, the
customer's XLIO build does not include uTLS ‚Äî TLS offload recommendations are not
applicable.

#### TSO enabled but no TSO packets **(TCP only)**
**WARNING:** `ring_tx_tso_packets: 0 # WARNING: TSO explicitly enabled but no TSO packets sent (payload may be below MSS threshold)`

**What it means:** The user explicitly enabled TSO (`hardware_features.tcp.tso.enable: enable`)
but no packets used TSO. TSO coalesces multiple TCP segments into one large send,
dramatically reducing CPU overhead for bulk transfers.

**What to check:**
- `nic_device` ‚Äî the NIC may not support TSO (firmware/driver limitation)
- Message size ‚Äî TSO only activates when the payload exceeds the MSS (typically ~1460 bytes
  for MTU 1500). If the application sends small messages (< MSS), TSO has nothing to coalesce.
- `hardware_features.tcp.tso.max_size` ‚Äî maximum TSO segment size

**Config fix:**
- If the NIC doesn't support TSO: nothing to fix, consider setting
  `hardware_features.tcp.tso.enable: auto` (default) to suppress the warning
- If message size is below MSS: TSO won't help ‚Äî this is expected behavior, not a problem
- Verify NIC firmware is up to date

**Ask the user:** "What is the typical message size your application sends?
Is it smaller than ~1460 bytes?"

---

#### TLS resyncs **(TCP only)**
**WARNING:** `ring_tls_rx_resyncs: <N> # WARNING: TLS RX resync (HW->SW fallback)`
**WARNING:** `ring_tls_tx_resyncs: <N> # WARNING: TLS TX resync (HW->SW fallback)`

**What it means:** TLS hardware offload had to fall back to software processing.
This happens when the NIC loses synchronization with the TLS record stream
(e.g., due to out-of-order packets, retransmits, or record boundary misalignment).
Each resync incurs a performance penalty.

**What to check:**
- `tx_retransmits` ‚Äî retransmits can cause TLS resyncs
- `hw_rx_packets_dropped` ‚Äî dropped packets cause RX resyncs
- Network stability ‚Äî packet reordering triggers resyncs

**Config fix:**
- Fix the underlying cause (retransmits, drops) using the rules above
- If resyncs are frequent despite a clean network: consider disabling HW TLS offload
  (`hardware_features.tcp.tls_offload.rx_enable: false` / `tx_enable: false`) ‚Äî
  software TLS may be more reliable on some firmware versions
- Note: `tls_offload.rx_enable` defaults to `false` ‚Äî if the customer sees RX resyncs,
  they explicitly enabled it. `tls_offload.tx_enable` defaults to `true`.

**Ask the user:** "Are you seeing retransmits or packet drops alongside the TLS resyncs?"

---

#### TLS authentication failure **(TCP only)**
**WARNING:** `ring_tls_rx_auth_fail: <N> # WARNING: TLS HW authentication failure`

**What it means:** The NIC's TLS hardware detected authentication failures (MAC
verification failed). This indicates corrupted TLS records ‚Äî a serious data
integrity issue.

**What to check:**
- NIC firmware version ‚Äî may be a known firmware bug
- Network path ‚Äî data corruption in transit (rare but possible with faulty cables/switches)

**Config fix:**
- Disable HW TLS RX offload (`hardware_features.tcp.tls_offload.rx_enable: false`)
  to fall back to software TLS (which will reject corrupted records safely)
- Update NIC firmware

**Ask the user:** "What NIC firmware version are you running? Has this been reported
to NVIDIA support?"

### Connection & Polling

#### Dropped connections **(TCP only)**
**WARNING:** `listen_conn_dropped: <N> # WARNING: connections dropped (backlog full?)`

**What it means:** Incoming TCP connections were dropped because the listen backlog
was full. The application wasn't calling accept() fast enough.

**What to check:**
- `listen_conn_established` vs `listen_conn_accepted` ‚Äî large gap means accept() is slow
- If `listen_conn_accepted: 0` with `listen_conn_established > 0` ‚Äî this is normal
  for event-driven APIs (poll groups / XLIO Ultra API) which don't use accept()
- Application architecture ‚Äî single-threaded accept() bottleneck?

**Config fix:**
- Not typically an XLIO config issue ‚Äî the application needs to accept connections faster
- Increase the application's listen backlog (the `backlog` argument to `listen()`)
- For nginx: ensure `applications.nginx.workers_num` matches available CPU cores

**Ask the user:** "What listen backlog size does your application use?
How many new connections per second do you expect?"

---

#### Low poll hit rate
**WARNING:** `poll_hit_rate: <pct>% # WARNING: low poll hit rate` (threshold: < 80%)

**What it means:** When XLIO polls the NIC for incoming data, it often finds nothing.
This means the application is polling more frequently than data arrives, or the
polling strategy is inefficient.

**Note:** If the report shows `poll_hit_rate: N/A` with a note about event-driven API,
this metric is not applicable ‚Äî the application uses poll groups / XLIO Ultra API
which bypasses the poll/recv loop. This is not a problem.

**What to check:**
- Is the workload naturally bursty? Low hit rates during idle periods are expected.
- `performance.polling.iomux.poll_usec` ‚Äî how long each poll call spins
- `performance.polling.iomux.poll_os_ratio` ‚Äî ratio of OS vs XLIO polls
- `profiles.spec` ‚Äî latency profiles use aggressive polling (100% CPU) which may
  show low hit rates under light load

**Config fix:**
- For throughput workloads: reduce `performance.polling.iomux.poll_usec` to lower
  CPU waste when no data is available
- For latency workloads: low hit rate is acceptable ‚Äî the CPU cost is the price
  of low latency. No change needed.
- Consider `performance.polling.rx_cq_wait_ctrl: true` (default: `false`) ‚Äî ensures
  file descriptors are added only to sleeping sockets' epoll descriptors, reducing
  kernel scan overhead when many sockets have low poll hit rates

**Ask the user:** "Is low latency or high throughput more important for this workload?
Is the traffic pattern steady-state or bursty?"

### No WARNINGs ‚Äî Throughput Below Expectations

Use this checklist when the report has no `# WARNING` annotations but the
customer reports throughput below their NIC's capacity, or latency above
expectations.

**Before tuning, always ask:**
- "What throughput/latency did you expect, and what did you measure?"
- "What benchmarking tool and parameters did you use?"
- "How long did the test run?" (if < 5 min, throughput includes startup overhead)

#### Checklist: MTU & Jumbo Frames
Check `nic_device` in System Context for the MTU value.
- MTU 1500 (default) limits throughput due to per-packet overhead.
- For bulk throughput workloads on a trusted network, recommend jumbo frames:
  `network.protocols.ip.mtu: 9000` (default: `0`, meaning use OS MTU).
  **Both endpoints and all switches must support the same MTU.**
- Ask: "What is the MTU on both endpoints and any intermediate switches?"

#### Checklist: TSO & LRO **(TCP only)**
Check the Effective Config for TSO and LRO settings.
- **TSO** (`hardware_features.tcp.tso.enable`, default: `auto`): Coalesces multiple
  TCP segments into one NIC send. Critical for TX throughput with large transfers.
  If not in Effective Config, it's on `auto` (enabled when NIC supports it).
  For explicit control: set to `enable`.
- **LRO** (`hardware_features.tcp.lro`, default: `auto`): Coalesces multiple received
  TCP segments into larger buffers. Reduces per-packet CPU cost on RX.
  If not in Effective Config, it's on `auto`. For explicit control: set to `enable`.
- If `ring_tx_tso_packets` is present and non-zero, TSO is working. If absent or 0
  with `tso.enable: auto`, TSO may not be activating ‚Äî check NIC support.

#### Checklist: Profile Selection
Check `profile_spec` in Active Profile.
- `none` ‚Äî no profile active. The customer is using raw defaults.
  - For HTTP proxy/load balancer: recommend `nginx` profile
  - For low-latency trading: recommend `latency` or `ultra_latency` profile
  - Profiles set dozens of config values optimally for the workload pattern
- If a profile is already active, the customer has the basics covered ‚Äî look deeper.

#### Checklist: Memory Headroom
Check `core.resources.memory_limit` in Effective Config.
- Default is 2 GB. For high-throughput workloads (especially with many connections
  or large ring sizes), this may be too low.
- Even without allocation failure WARNINGs, the pools may be sized smaller than
  optimal because memory_limit caps them.
- Recommend 4-8 GB for 100 Gbps workloads, 8-16 GB for 200+ Gbps.

#### Checklist: Ring Sizes
If memory is sufficient but throughput is below expectations:
- `performance.rings.tx.ring_elements_count` (default: `32768`): Larger TX ring
  absorbs send bursts without stalling.
- `performance.rings.rx.ring_elements_count` (default: `32768`, but **`128` when
  striding RQ is enabled, which is the default**): Larger RX ring absorbs receive bursts.
- `performance.rings.rx.spare_buffers` (default: `32768`, but **`128` when striding
  RQ is enabled, which is the default**): Extra buffers for burst absorption.

#### Checklist: CPU Affinity & Isolation
The report doesn't show CPU utilization ‚Äî this is the most common blind spot.
- Ask: "Is the application pinned to specific CPU cores? Are those cores isolated
  from OS interrupts (`isolcpus`, `irqbalance`)?"
- Ask: "What CPU utilization do you see during the test? If cores are at 100%,
  XLIO config changes won't help ‚Äî the bottleneck is CPU."

#### Checklist: Polling Configuration
If the workload is latency-sensitive:
- `performance.polling.iomux.poll_usec` (default: `100000` ¬µs = 100ms):
  For ultra-low latency, set to `-1` (infinite polling, 100% CPU).
- `performance.polling.blocking_rx_poll_usec` (default: `100000` ¬µs):
  Same consideration for blocking recv operations.
- The `latency` and `ultra_latency` profiles set these automatically.

If the workload is throughput-sensitive:
- Lower polling duration and let XLIO use interrupts for idle periods.
- Enable `performance.polling.rx_poll_on_tx_tcp: true` (default: `false`) for
  bidirectional workloads ‚Äî polls RX during TX operations.

#### Checklist: Peer Configuration
- Ask: "Is the other end also running XLIO? Can you share its tuning report?"
- Mismatched MTU between endpoints causes fragmentation or drops.
- If the peer is using kernel networking, XLIO-side tuning has limited effect
  on end-to-end throughput.

poll hit rate, retransmit rate, or offload traffic split. The throughput numbers
are available but we're flying partially blind.

**Recommendation:** Before any tuning analysis, re-run with per-socket stats enabled:
```
{
  "monitor": {
    "stats": {
      "fd_num": 100  // was: 0 (default) ‚Äî set to at least total socket count
    }
  }
}
```
This will produce a full-mode report with all 21 possible WARNINGs and per-socket
diagnostics.

**Questions to ask the customer:** "How many concurrent sockets does your application
typically use? Set `monitor.stats.fd_num` to at least that number and re-run."

**Report excerpt:**
```
## Socket Summary

**Analysis:** This is a **UDP-only workload** (`tcp_sockets: 0`). All TCP-specific

The Effective Config shows all defaults ‚Äî `core.resources.memory_limit` is at 2 GB.

**Recommendation:**

```

**Analysis:** This is a healthy report from an application using XLIO Ultra API /

## After Applying Changes

When a customer applies your recommendations and shares a new report, compare
the two reports systematically:

### What to Compare
1. **Check Effective Config** ‚Äî verify the recommended changes actually took effect.
   Check the `reason:` field: your recommended changes should appear as
   `reason: User-configured`. If a recommended knob doesn't appear in the new
   Effective Config section, the customer may not have applied it correctly.
   Common causes:
   - Typo in config knob path
   - Wrong config file (check `XLIO_CONFIG_FILE` or `XLIO_INLINE_CONFIG` env var)
   - Missing `XLIO_USE_NEW_CONFIG=1` ‚Äî this environment variable is **required** to
     activate the JSON config system. Without it, XLIO uses legacy `XLIO_*` environment
     variables and the JSON config file is ignored entirely. If the Effective Config
     section shows `# Config registry not available`, this is almost certainly the cause.
   - A profile overriding the value ‚Äî if a knob shows `reason: Profile` instead
     of `reason: User-configured`, the profile took precedence. The customer may
     need to add the override after the profile setting, or switch profiles.

2. **Check if WARNINGs resolved** ‚Äî if the original report had WARNINGs, check
   whether they disappeared, reduced in count, or new ones appeared.
   - WARNING gone ‚Üí fix worked
   - WARNING count reduced ‚Üí partial improvement, may need further tuning
   - New WARNINGs appeared ‚Üí the fix may have shifted the bottleneck (e.g.,
     increasing memory_limit resolved buffer failures but now WQE exhaustion
     appears because more buffers means more concurrent sends)

3. **Compare throughput** ‚Äî if both reports have throughput values, compare them.
   Ensure both runs were similar duration (short runs have startup overhead).

4. **Check for shifted bottlenecks** ‚Äî fixing one issue often reveals the next
   bottleneck. For example:
   - Fixing buffer pool failures ‚Üí now HW drops appear (ring too small)
   - Fixing HW drops ‚Üí now SW drops appear (application too slow)
   - Fixing SW drops ‚Üí clean report but CPU at 100% (CPU-bound)

### Comparing Sender and Receiver Reports
When the customer shares reports from both endpoints, cross-reference them:

| Sender Report | Receiver Report | Likely Cause |
|---|---|---|
| `tx_retransmits` high | `hw_rx_packets_dropped` high | Receiver NIC can't keep up ‚Äî increase receiver ring sizes and memory |
| `tx_retransmits` high | Clean (no drops) | Network issue ‚Äî switch congestion, cable fault, or MTU mismatch |
| Clean | `sw_rx_packets_dropped` high | Receiver application too slow ‚Äî CPU-bound or polling misconfigured |
| `nic_device` MTU: 9000 | `nic_device` MTU: 1500 | MTU mismatch ‚Äî causes fragmentation or drops. Align MTU on both ends. |

**Key check:** Compare `nic_device` MTU values between the two reports. Mismatched
MTU is the #1 cross-endpoint configuration error.

### Common Follow-Up Patterns
- "I increased memory_limit but still see failures" ‚Üí check if hugepages were
  also increased at the OS level
- "WARNINGs are gone but throughput didn't improve" ‚Üí use the "No WARNINGs"
  checklist
- "New WARNINGs appeared" ‚Üí address the new bottleneck using the troubleshooting
  rules

## Report Limitations

Things the report does NOT tell you ‚Äî the chatbot must be aware of these gaps
and compensate by asking the customer:

1. **No per-socket breakdown.** Traffic stats are aggregated across all sockets.
   You cannot identify which specific socket or connection is problematic.
   Ask: "Can you narrow down which connections have issues?"

2. **No timeline data.** All counters are cumulative for the process lifetime.
   You cannot distinguish between "steady 1% error rate" and "100% errors for
   the first 5 seconds then clean" ‚Äî both look the same.
   Ask: "Did the issue happen throughout the test, or only at startup?"

3. **Lifetime-average throughput.** The `rx_throughput` / `tx_throughput` values
   include startup overhead (device probing, memory registration, TCP slow-start).
   For short runs (< 5 min, the report annotates this with
   `# Note: lifetime average, process ran < 5min`), steady-state throughput is
   likely higher than reported. Recommend running > 5 minutes for accurate benchmarks.

4. **No peer-side visibility.** TX retransmits may be caused by the receiver
   dropping packets, but the report only shows the sender's view.
   Ask: "Can you share the peer's tuning report?"

5. **No CPU utilization.** The report doesn't show whether the application is
   CPU-bound. A "clean" report with low throughput may indicate a CPU bottleneck
   that no XLIO config change can fix.
   Ask: "What CPU utilization do you see during the test?"

6. **Counter overflow risk (actionable).** Per-socket counters use uint32_t
   (`n_rx_packets`, `n_tx_sent_pkt_count`, `n_tx_retransmits`, etc.). At 100 Gbps
   with 64-byte packets, `n_rx_packets` overflows in ~28 seconds. The report's
   aggregation uses uint64_t, but individual socket counters may have already
   wrapped before aggregation.
   **How to detect:** Check `total_rx_bytes / total_rx_packets`. If the average
   packet size is outside the 64‚Äì9000 byte range, counter wraparound is likely.
   Similarly, if `tx_retransmit_rate` shows an impossibly high value (> 50%),
   suspect uint32 overflow in per-socket retransmit counters.
   **Mitigation:** For long-running high-rate processes, recommend shorter
   benchmark runs (5-30 min) or periodic report generation to avoid overflow.

7. **No network topology.** The report doesn't know about switches, routers,
   or MTU mismatches along the path. MTU is shown for local NICs only.
   Ask: "What is the MTU on the peer side and intermediate switches?"

8. **Partial stats coverage.** When `monitor.stats.fd_num` is smaller than the
   actual socket count, per-socket traffic stats only cover a subset of sockets.
   The report annotates this with:
   `# Note: per-socket traffic stats cover X/Y sockets`
   Traffic split numbers (offloaded vs non-offloaded bytes) are **underestimates**.
   Recommend increasing `monitor.stats.fd_num` to at least Y.
